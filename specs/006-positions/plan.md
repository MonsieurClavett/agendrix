# Implementation Plan: Positions

**Branch**: `006-positions` | **Date**: 2026-05-28 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/006-positions/spec.md`

## Summary

Add a `Position` entity scoped per `Company`. Manager has a new `/positions`
page for CRUD. `Shift` gains an optional `positionId` FK (ON DELETE
SET NULL). The shift form gains a position picker. Calendar cards render
a colored left-border accent + the position's name (replacing "Quart").
The Phase 4 filter panel becomes functional: position checkboxes filter
the grid, "Sans position" toggles untagged shifts, and the "Gérer par
Position" segment switches the grid from per-employee rows to per-position
rows. Drag-and-drop reassigns position in the new grouping mode (and
keeps it in employee mode).

One Prisma migration. One new repository file. One new Server-Action
folder. One new route (`/positions`). Several existing components
extended.

## Technical Context

**Language/Version**: TypeScript (strict) on Next.js 16 (App Router), Node.js 24 — carry-over.

**Primary Dependencies**: no new deps. Everything reuses the Phase 0–4 stack (Prisma, Auth.js, shadcn primitives, dnd-kit, sonner).

**Storage**: Same Neon Postgres. One new migration `add_positions`:
1. Create `Position` table.
2. Add nullable `positionId` column to `Shift` with FK to `Position(id)` ON DELETE SET NULL.

**Testing**: Manual browser smoke (carry-over per Constitution III).

**Target Platform**: Web — same as Phase 4.

**Project Type**: Single Next.js project. New files under
`src/lib/repositories/position.ts`, `src/actions/positions/`,
`src/app/(dashboard)/positions/`. Existing components extended in
place (`ShiftBlock`, `ShiftDialog`, `ScheduleCalendar`, `FilterPanel`).

**Performance Goals**:
- Position CRUD page renders < 200 ms for ≤ 50 positions.
- Filter checkboxes update the grid in < 50 ms client-side for ≤ 50 shifts.
- Drag-end with position update completes in < 500 ms (server round-trip).

**Constraints**:
- All Phase 0–4 invariants carry over: tenant isolation, role gating,
  overlap detection, theme persistence, totals correctness.
- Position deletion MUST be non-cascading on shifts (FR-005).
- Cross-tenant `positionId` injection MUST be rejected server-side (FR-009).
- Drag in "Gérer par Employé" mode MUST NOT change `positionId` (FR-012).
- Drag in "Gérer par Position" mode MUST update `positionId` to the
  destination row (FR-018).

**Scale/Scope**: ≤ 10 positions per company, ≤ 50 shifts / week — same as Phase 2/4.

**Structure Decision**: The schedules page introduces a new client
wrapper component, `ScheduleView`, that owns the filter and grouping
state for Phase 5. The page (Server Component) fetches the data and
passes it down once. `ScheduleView` renders both `FilterPanel` and
`ScheduleCalendar`, lifting checkbox state up via callbacks. This
avoids URL state management for Phase 5 (deferred per spec) and keeps
each component small.

`Position` colors are a fixed enum in `src/lib/positions.ts` (8 OKLCH
pairs `{ bg, fg, accent }`). Storing the position's `color` field as
the palette key (a short enum-like string, e.g. `"teal"`, `"coral"`)
means the rendering code maps the key to the OKLCH triplet — no raw
hex stored, no AA contrast surprises.

In "Gérer par Position" mode, the `DropCell` id format changes from
`${day}|${employeeId}` to `${day}|pos:${positionId|none}`. The
`onDragEnd` handler inspects the id prefix to know which mode applied
and dispatches an update that changes either employeeId+date (employee
mode) or positionId+date (position mode), keeping the other field
intact.

## Post-Design Re-Check

After Phase 1 design:

- The position deletion → shift orphaning is handled by Prisma's
  `ON DELETE: SetNull` on the FK — no application-level cleanup needed.
- The "Gérer par Position" drag preserves overlap semantics: overlap
  is checked per-employee (FR-018), so the existing
  `WHERE employeeId = ? AND startsAt < newEnd AND endsAt > newStart`
  query is sufficient — no Phase 5 change to the overlap logic.
- The FilterPanel becomes truly stateful but it's still per-page
  session (no persistence) — minimal blast radius.
- Position assignment is guarded server-side: `updateShift` /
  `createShift` accept `positionId` and verify
  `ctx.companyId === position.companyId` inside the transaction.
  Cross-tenant injection is rejected with `POSITION_NOT_FOUND`.

Gate remains ✅ PASS.
