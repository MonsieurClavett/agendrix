Phase 1 of `/speckit-plan`. Tiny schema diff on the existing `Shift`
table.

## Enum: `ShiftStatus`

```prisma
enum ShiftStatus {
  DRAFT
  PUBLISHED
}
```

## `Shift` (extended)

| Field      | Type           | Notes                              |
|------------|----------------|------------------------------------|
| (existing) | …              | unchanged                          |
| `status`   | `ShiftStatus`  | NOT NULL, DEFAULT `DRAFT`          |

Plus a new composite index `(companyId, status)` to keep both the
EMPLOYEE read (filter on status) and the publish query (count + bulk
update on DRAFTs) cheap.

## State transitions

```
                  +-----------+
  create  ------> |   DRAFT   |
                  +-----+-----+
                        |
                        | publishWeek / unpublish (MANAGER only)
                        v
                  +-----------+
                  | PUBLISHED |
                  +-----------+
```

- `DRAFT → PUBLISHED` via `publishWeek` (bulk) or any future `publishShift` (none planned).
- `PUBLISHED → DRAFT` via `unpublishShift` (single).
- Updates and deletes do NOT change status.
- Drag-and-drop (an `updateShift`) does NOT change status.

## Migration

One Prisma migration `add_shift_status`:

```sql
-- 1. Create enum
CREATE TYPE "ShiftStatus" AS ENUM ('DRAFT', 'PUBLISHED');

-- 2. Add column with DRAFT default (Prisma generates this)
ALTER TABLE "Shift"
  ADD COLUMN "status" "ShiftStatus" NOT NULL DEFAULT 'DRAFT';

-- 3. Backfill: every existing row becomes PUBLISHED
UPDATE "Shift" SET "status" = 'PUBLISHED';

-- 4. New index
CREATE INDEX "Shift_companyId_status_idx" ON "Shift"("companyId", "status");
```

Step 3 makes pre-Phase-8 shifts continue to be visible to their
employees. The default `DRAFT` applies only to NEW rows.

The migration is generated by `prisma migrate dev --create-only`,
then we hand-edit the SQL file to insert the `UPDATE` statement
between step 2 and step 4. (Prisma cannot express a backfill
declaratively.)
